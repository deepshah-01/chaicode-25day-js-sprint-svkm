# 🚀 Super 20: 25-Day JavaScript Production Floor

Welcome to the central nervous system of the **Super 20 JavaScript Sprint**.

This is not a typical college project repository. This is a simulated startup engineering environment. For the next 25 days, 20 elite builders are escaping tutorial hell by writing raw JavaScript, pushing code daily, handling merge conflicts, and peer-debugging in public.

Powered by the **Chai aur Code** ecosystem.

---

## 🏗️ The Dual-Repo Architecture

We operate on a two-repo system to build both your personal brand and your team-collaboration skills:

### 1. Your Personal Repo (The Sandbox)

You must maintain a separate personal GitHub repository for the challenge and push your work there before contributing to the production repository.

This is your personal proof of work and helps maintain your GitHub contribution graph.

### 2. This Central Repo (The Production Floor)

After updating your personal repository, you will contribute your final daily code here.

This simulates a real engineering environment where multiple developers collaborate on the same codebase, manage branches, review code, and resolve conflicts.

---

## 📂 Directory Structure Rules (Mandatory)

With 20 developers contributing to the same repository daily, chaos is inevitable without strict structure.

### Rules

- Do **NOT** place files in the root directory.
- Create a folder using your exact name.
- Create a separate folder for each day inside your directory.
- Keep all project files, notes, and assets inside the corresponding day's folder.
- Do **NOT** modify another student's directory.

### Example Structure

```text
📦 super20-js-production-floor

 ┃ ┃ ┣ 📜 index.html

```

## 🌿 Branch Workflow

### Create Your Branch

Create a branch using your name:

```bash
git checkout -b your-name
```

### Keep Your Branch Updated

```bash
git pull origin main
```

### Commit Your Changes

```bash
git commit -m "Day XX: Added [Topic Name] - [Your Name]"
```

### Push Your Branch

```bash
git push origin your-name
```

### Open a Pull Request

- Create a Pull Request against `main`.
- Wait for review approval.
- Resolve requested changes if any.
- Merge only after approval.

## 🛑 Rules of Survival

### Zero Tolerance for Ghosting

Missing a daily push, Pull Request, or the 10:00 PM sync without a valid emergency may result in immediate removal from the program.

### Never Push Directly to Main

All work must go through:

1. Branch
2. Pull Request
3. Review
4. Merge

### Don't Touch Other Folders

You are only authorized to modify files inside your own directory.

Example:

```text
YourName/
```

### Fix Your Own Conflicts

If your changes cause merge conflicts:

- Investigate the issue.
- Resolve conflicts locally.
- Push the fix.
- Ask for help during the 10:00 PM sync if needed.

Learning to resolve conflicts is part of the challenge.

### Keep Commits Meaningful

Use clear commit messages.

Good:

```bash
git commit -m "Day 05: Functions and Scope - John Doe"
```

Bad:

```bash
git commit -m "update"
```
